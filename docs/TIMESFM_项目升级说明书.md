# TimesFM 项目升级说明书

## 1. 文档目的

本文档用于说明如何将当前项目从“新闻 + 价格解释系统”升级为“研究、预测、风控一体化市场研究平台”，重点围绕 TimesFM 2.5 及其后续 few-shot / ICF 路线展开。

本文档覆盖以下内容：

- 当前项目已经具备的能力
- TimesFM 与最新论文的研究结论
- 适合本项目的升级方案
- 当前阶段可以立即实现的功能
- 中长期可演进能力
- 所需系统条件、依赖与部署要求
- 技术可行性、业务可行性与主要风险研判

## 3. 当前系统的主要短板

虽然当前项目在事件解释、结构化研究和风险提示方面已较完整，但仍存在一个关键缺口：

### 3.1 缺少概率化预测层

当前系统强于：

- 解释为什么涨跌
- 总结过去 24 小时主线
- 提示风险点和失效条件

但相对欠缺：

- 未来 30 分钟 / 120 分钟价格延续概率
- 区间式预测而非单一方向判断
- 概率化的不确定性评估
- 量化风险带而不只是文字风险提示

### 3.2 缺少统一的时间序列基础模型层

当前价格分析是基于：

- 实际价格变化
- 历史事件后续表现
- 规则判断
- 新闻证据归因

但还没有引入一个专门负责“未来路径预测”的基础模型层。

这正是 TimesFM 最适合补上的位置。

## 4. TimesFM 研究结论

### 4.1 TimesFM 的定位

TimesFM 是 Google Research 推出的时间序列基础模型，核心目标是让模型在不做任务级微调的情况下，直接对不同领域、不同频率、不同长度的时间序列进行零样本预测。

其核心价值在于：

- 零样本预测能力强
- 可适应不同上下文长度和预测长度
- 适合作为统一的时间序列预测底座

### 4.2 TimesFM 基础论文要点

基础论文《A decoder-only foundation model for time-series forecasting》的主要结论是：

- TimesFM 采用 decoder-only patched transformer 架构
- 预训练数据规模非常大
- 在大量未见过的数据集上可实现接近 SOTA supervised 模型的效果
- 重点优势是泛化能力和零样本能力

对本项目的意义：

- 不需要为每个资产单独训练一个预测模型
- 可以先做统一的时间序列预测层
- 对多市场、多资产项目尤其适合

### 4.3 TimesFM 2.5 的最新能力

根据官方仓库说明，TimesFM 2.5 相比旧版本有以下升级：

- 参数量从 500M 缩小到 200M
- context length 扩大到 16k
- 支持连续分位数预测
- 支持更长 horizon
- 不再依赖旧频率指示器
- 新增 covariates 支持

对本项目的直接价值：

- 适合长历史窗口输入
- 适合输出区间预测而不是只输出点预测
- 适合结合新闻、跨资产、事件标签等协变量

### 4.4 最新论文方向：ICF / few-shot

最新路线《In-Context Fine-Tuning for Time-Series Foundation Models》的核心思想是：

- 不再只输入目标序列
- 同时把若干“相关时间序列样本”一起作为上下文
- 让模型在推理时利用 support examples 学习
- 获得接近 supervised fine-tuning 的效果

其对本项目最大的启发不是“立刻替换模型”，而是：

- 你现在已经有“相似历史事件检索”
- 已有“跨资产联动”
- 已有“主线归因”

这些能力天然可以演进为 future support set 选择器。

## 5. 为什么 TimesFM 适合本项目

### 5.1 当前项目已具备高质量价格序列入口

TimesFM 需要可靠时间序列输入，而本项目已具备：

- 分钟级价格数据读取
- 不足数据 fallback
- 交易所回退加载
- 多市场、多代码候选适配

这意味着价格输入层已经成熟，不需要从零建设。

### 5.2 当前项目已具备丰富协变量来源

TimesFM 2.5 支持 covariates，本项目天然拥有比一般价格系统更丰富的协变量：

- 直接新闻数量
- 驱动新闻数量
- 新闻重要性评分
- 宏观事件标签
- 跨资产变动
- 当前场景路由标签
- 当前资产类别
- 风险状态

这使得本项目非常适合做“价格 + 事件 + 跨资产 + 结构化标签”的协变量预测。

### 5.3 当前项目已具备 ICF 路线的 support 检索基础

你当前系统已经能找到：

- 相似历史事件
- 同类主线样本
- 当前主线下的历史新闻反应样本

这意味着未来升级到 few-shot / ICF 时，不需要重新建设 support 检索层。

### 5.4 当前项目最适合的是“预测增强”，不是“重做内核”

TimesFM 最适合的方式是作为现有研究系统的预测增强层，而不是替代：

- 新闻研究
- 风险经理
- 场景路由
- 历史主线系统

这样做：

- 改动更可控
- 风险更小
- 价值更快落地

## 6. 升级目标架构

推荐将项目升级为三层结构：

### 6.1 第一层：研究解释层

保留现有能力：

- 新闻检索
- 经济数据分析
- 地缘政治分析
- 技术与缠论分析
- 多空辩论
- 研究经理裁决
- 风险经理文本判断

### 6.2 第二层：TimesFM 预测层

新增统一时间序列预测能力：

- 点预测
- 分位数预测
- 波动区间
- 方向延续概率
- 反转概率

### 6.3 第三层：融合决策层

将预测层与研究层融合：

- 新闻强弱 + 价格预测
- 主线归因 + 延续概率
- 风险经理 + 分位区间
- 场景路由 + regime 判断

最终输出：

- 当前主线
- 当前方向偏好
- 风险区间
- 最不适合做的动作
- 等待确认条件

## 7. 当前阶段可以立即实现的功能

### 7.1 零样本价格预测

可优先支持以下频率：

- 5m
- 15m
- 1h

可输出：

- point forecast
- quantile bands
- continuation probability
- reversal risk

### 7.2 实时关注预测增强

可在实时关注结果中新增字段：

- `forecast_30m`
- `forecast_120m`
- `forecast_quantiles`
- `forecast_confidence`
- `forecast_regime`

升级后实时关注不只回答：

- 为什么价格动了

还回答：

- 接下来延续的概率多大
- 风险带有多宽
- 当前更适合追随还是等待

### 7.3 历史分析预测增强

在历史事件分析中新增：

- 事件触发时的预测中位路径
- 事件触发时的预测区间
- 预测与实际偏差
- surprise score

这样可以把“是否已被市场消化”从经验判断升级为：

- 历史样本对比
- 模型预测区间对比
- 实际表现偏离程度

### 7.4 风险经理量化增强

在风险经理规则摘要中增加：

- 下分位风险
- 上下分位差
- 波动区间宽度
- 高不确定性警告

这样风险经理会从文字型风险提示升级为：

- 结构化风险提示
- 概率化风险提示

### 7.5 场景化路由增强

当前场景路由可再引入预测信号：

- `forecast_followthrough_strong`
- `forecast_dispersion_high`
- `forecast_reversal_risk_high`

用于指导系统选择：

- 趋势延续路径
- 风险防守路径
- 均衡观察路径

### 7.6 金十新闻事件后续预测

金十新闻进入系统后，实时关注可以直接输出：

- 新闻触发后的短线价格预测
- 未来 30m / 120m 延续概率
- 若走势不延续，最先失效的信号

这是当前系统最有业务价值的首批落地场景之一。

## 8. 中长期可升级点

### 8.1 Covariates 版本

这是第一优先级中期增强项。

建议接入的协变量包括：

#### 动态数值协变量

- 最近收益率
- ATR
- realized volatility
- 5m / 30m 波动幅度
- 新闻数量
- 新闻 importance 均值
- 跨资产变化幅度

#### 动态类别协变量

- 时段标签
- 星期标签
- 宏观事件日标签
- 场景路由标签

#### 静态协变量

- 市场类型
- 资产类别
- 风险属性

### 8.2 Support Set 检索层

未来要做 ICF 风格 few-shot，需要支持样本选择器。

本项目天然适合做三类 support：

- 同资产过去相似事件
- 同主线下的历史事件
- 高相关跨资产历史窗口

### 8.3 ICF / few-shot 预测

未来可升级为：

- 检索 support windows
- 构建 in-context examples
- 做 few-shot forecast

但这一步建议排在二期或三期，而不是首期。

### 8.4 模型评估与反馈闭环

后续建议建立 TimesFM 预测评估体系：

- 方向命中率
- quantile coverage
- calibration
- event continuation hit rate
- risk signal hit rate

### 8.5 预测缓存与任务调度

中长期建议加入：

- 热门资产预测预计算
- 定时刷新预测缓存
- 事件触发增量刷新
- 页面优先读取缓存

## 9. 推荐的首期落地方案

### 9.1 MVP 定位

首期不建议做“大而全”的预测平台，而建议做：

**事件后延续概率引擎**

### 9.2 MVP 输入

- 最近 N 根 5m / 15m / 1h bars
- 当前直接新闻强度
- 驱动新闻强度
- 关键跨资产变化
- 当前场景路由标签

### 9.3 MVP 输出

- 未来 30m 中位数方向
- 未来 120m 中位数方向
- 10 / 50 / 90 分位区间
- continuation probability
- reversal risk
- forecast dispersion

### 9.4 MVP 接入点

- 实时关注
- 历史分析
- 风险经理

### 9.5 MVP 不做的事

首期不建议做：

- 全量 ICF
- 在线微调
- 主服务同步重推理
- 把 TimesFM 直接当交易信号引擎

## 10. 系统条件

### 10.1 Python 与项目依赖条件

当前项目声明 Python 版本要求为：

- Python >= 3.11

当前项目依赖尚未包含：

- torch
- huggingface_hub
- safetensors
- jax
- jaxlib
- scikit-learn

### 10.2 TimesFM 接入所需新增依赖

#### 基础推理依赖

- `torch>=2.0`
- `huggingface_hub`
- `safetensors`
- `timesfm` 源码安装或等效安装方式

#### 若使用 covariates

- `jax`
- `jaxlib`
- `scikit-learn`

### 10.3 数据条件

需要满足：

- 分钟级 K 线数据连续度足够
- 新闻与价格时间对齐稳定
- 跨资产参考序列可获取
- 节假日 / 停牌 / 数据缺口可被预处理

### 10.4 硬件条件

#### 最低可运行条件

- CPU 环境可运行
- 适合少量资产、离线分析、低频任务

#### 推荐生产条件

- 单独预测 worker
- 8 核以上 CPU
- 32GB 内存更稳
- 若高频多资产推理，建议 GPU

### 10.5 部署条件

不建议直接把 TimesFM 推理塞进当前 Flask 主线程。

推荐部署形态：

- 主应用负责页面、研究、任务编排
- 独立预测服务负责 TimesFM 推理
- 主应用通过缓存或内部接口获取预测结果

## 11. 技术可行性研判

### 11.1 总体结论

将本项目升级为 TimesFM 增强型研究平台，技术上可行，且首期落地难度可控。

### 11.2 可行性分级

#### 阶段 1：TimesFM 2.5 零样本接入

- 可行性：高
- 原因：
  - 现有价格序列入口已具备
  - 现有实时关注和历史分析入口已具备
  - 只需增加预测层与缓存层

#### 阶段 2：Covariates 接入

- 可行性：高
- 原因：
  - 项目已有丰富协变量
  - 与业务语义高度匹配

#### 阶段 3：Support Retrieval

- 可行性：中高
- 原因：
  - 已有相似历史事件检索
  - 已有跨资产观察
  - 已有主线标签

#### 阶段 4：ICF 工程化

- 可行性：中
- 原因：
  - 论文方向先进
  - 但实际工程成熟度、部署复杂度、评估成本更高

### 11.3 最大优势

本项目接入 TimesFM 的最大优势不是价格数据本身，而是：

- 事件数据丰富
- 新闻结构化较强
- 支持跨资产联动
- 已有相似历史事件检索
- 已有研究经理和风险经理链路

这意味着本项目不是普通的时间序列接入场景，而是一个非常适合做“预测增强研究系统”的场景。

### 11.4 最大风险

#### 风险 1：把零样本预测误当交易信号

缓解方式：

- 先用于风险过滤、方向确认和概率判断
- 不直接作为买卖信号

#### 风险 2：高频金融噪声太强

缓解方式：

- 首期重点做 30m / 120m 延续概率，而不是 tick 级预测
- 配合分位区间使用，而不是只看 point forecast

#### 风险 3：推理拖慢主服务

缓解方式：

- 独立服务化
- 缓存化
- 异步化

#### 风险 4：ICF 过早上马导致复杂度失控

缓解方式：

- 首期只做 TimesFM 2.5 零样本与协变量版
- 将 ICF 放在二期或三期

## 12. 业务可行性研判

### 12.1 业务价值高

升级后系统将从：

- 解释系统

升级为：

- 解释 + 预测 + 风控系统

直接价值体现在：

- 用户看到新闻后，知道“后续大概率怎么走”
- 用户看到价格异动后，知道“这波是否值得追”
- 用户看到风险经理提示后，知道“哪里最容易失效”

### 12.2 与现有能力协同性强

TimesFM 不会和当前系统冲突，反而会增强：

- 实时关注
- 历史分析
- 场景化路由
- 风险经理
- 研究经理裁决

### 12.3 不会破坏现有体验

采用独立预测层方式后：

- 不需要推翻现有功能
- 不需要替换现有研究逻辑
- 只是在原有输出上增加“概率层”

因此业务可行性判断为高。

## 13. 推荐实施计划

### 阶段 A：首期升级

目标：

- 建立 TimesFM 2.5 零样本预测服务
- 先接 5m / 15m / 1h 三个频率
- 输出 point + quantile forecast
- 接入实时关注和历史分析

交付物：

- 独立预测服务

### 阶段 B：协变量增强

目标：

- 引入新闻、跨资产、场景路由协变量

### 阶段 C：support retrieval

## 16. 附录 A：量化研究设计与评估规范

这一部分用于把当前说明书从“系统升级方案”补充为“可进入量化研发排期的研究方案”。

### 16.1 先定义研究目标，而不是先定义模型

在量化研究里，必须先明确预测目标与交易目标，再讨论 TimesFM 是否适合。

建议首期研究目标定义为：

- `y_30m = t 时点到 t+30m 的对数收益率`
- `y_120m = t 时点到 t+120m 的对数收益率`
- `label_up/down/neutral = 基于未来收益率阈值映射的方向标签`

建议首期不把模型输出直接转成买卖信号，而分为三层用途：

- 研究用途：解释事件后的价格延续概率
- 风控用途：识别高不确定性与高回撤风险
- 信号用途：仅在评估通过后逐步开放

### 16.2 建议的预测任务拆分

建议至少拆分三类任务，而不是只做单一预测：

#### 任务 1：方向分类

目标：

- 预测未来 30m / 120m 上涨、下跌或中性

价值：

- 适合接入实时关注与场景路由

#### 任务 2：收益率回归

目标：

- 预测未来 30m / 120m 的收益率中位数

价值：

- 适合做主线强弱判断

#### 任务 3：分位数预测

目标：

- 预测未来收益率的 10 / 50 / 90 分位

价值：

- 适合风险经理和仓位控制

### 16.3 必须有基准模型

为了判断 TimesFM 是否真的有研究增益，必须建立 baseline。

建议至少包含以下对照组：

- 随机游走基线
- 持续动量基线
- 均值回复基线
- 线性回归基线
- XGBoost 或 LightGBM + 手工特征基线
- 当前规则系统基线

只有当 TimesFM 在这些基线之上带来稳定增益时，才值得进入生产链路。

### 16.4 评估指标必须分三层

#### 第一层：预测指标

- MAE
- RMSE
- sMAPE
- Directional Accuracy
- Brier Score
- Quantile Coverage
- Calibration Error

#### 第二层：事件指标

- 金十新闻后 30m 命中率
- 金十新闻后 120m 命中率
- CPI / 非农 / 央行事件日单独表现
- 突发地缘事件单独表现

#### 第三层：交易指标

- 胜率
- 平均盈亏比
- 最大回撤
- Sharpe
- Sortino
- Turnover
- 手续费后收益

### 16.5 评估必须采用时间滚动而不是随机切分

建议采用：

- Train：历史较长窗口
- Validation：后续连续时间窗口
- Test：最近连续时间窗口
- Walk-forward：逐步滚动重评估

不建议使用随机切分，因为金融时间序列存在明显的时序依赖与 regime 变化。

### 16.6 评估必须做 regime 分层

TimesFM 在不同市场状态下的效果可能差异很大，建议分层评估：

- 高波动 vs 低波动
- 事件日 vs 非事件日
- 趋势市 vs 震荡市
- 亚洲时段 vs 欧洲时段 vs 美盘时段
- 新闻密集时段 vs 平静时段

只有分层后，才能知道模型到底在哪些场景有效。

### 16.7 研究结论输出标准

建议首期评估通过标准不是单一收益，而是组合条件：

- 方向准确率显著优于 baseline
- Quantile coverage 基本稳定
- 高不确定性区间能有效过滤差交易
- 事件后预测在主要资产上具备一致增益

## 17. 附录 B：交易可执行性与决策矩阵

这一部分用于解决“预测得准，不等于能交易”的问题。

### 17.1 先区分研究输出、风控输出、交易输出

建议把模型输出分为三类：

#### 研究输出

用于：

- 事件解释
- 主线延续判断
- 反事实复盘

#### 风控输出

用于：

- 判断是否适合追随
- 判断是否应该减仓
- 判断是否属于高不确定性阶段

#### 交易输出

仅在满足严格条件下使用：

- 预测方向明确
- 分位区间较窄
- 风险经理未给出高风险
- 研究链路与跨资产链路一致确认

### 17.2 决策矩阵建议

建议把融合后的输出固化为规则矩阵，而不是纯文本判断。

#### 场景 A：新闻强 + 预测强 + 风险低

动作建议：

- 允许进入“延续跟踪”
- 提高研究优先级
- 可进入轻仓试错池

#### 场景 B：新闻强 + 预测弱 + 分歧大

动作建议：

- 不追价
- 只观察二次确认
- 风险经理优先级上升

#### 场景 C：价格强 + 新闻弱 + 跨资产不确认

动作建议：

- 判断为高假突破风险
- 降低趋势路径权重
- 等待更多证据

#### 场景 D：预测方向明确但下分位很差

动作建议：

- 允许方向判断进入研究层
- 禁止高杠杆
- 仓位受风险带控制

#### 场景 E：预测分位区间极宽

动作建议：

- 标记为高不确定性
- 只保留解释与观察用途
- 不进入交易信号层

### 17.3 交易可实现性约束

首期必须明确哪些场景只做研究，不做交易：

- 金十突发新闻发布后的第一分钟
- 低流动性时段的 5m 高频信号
- 点差明显扩大的时段
- 重大事件发布瞬间的极端波动阶段

建议首期只在以下方向考虑交易化评估：

- EURUSD
- XAUUSD
- BTCUSDT

原因：

- 流动性相对更好
- 宏观事件链路更清晰
- 新闻与价格联动更容易解释

### 17.4 不确定性的使用原则

分位区间不是展示项，而是风险约束变量。

建议规则：

- 分位区间过宽：只观察，不行动
- 中位数方向明确但尾部风险大：降低仓位
- 概率未经校准：不可直接映射成仓位
- 预测分歧高：风险经理优先级提升

### 17.5 交易动作必须独立于模型营销叙事

需要在文档中明确：

- “模型先进”不能作为放宽风控的理由
- “论文效果好”不能替代本项目回测结果
- “预测看起来很合理”不能替代严格执行矩阵

## 18. 附录 C：数据时序一致性、泄露控制与市场分层

这一部分是研究系统最容易被忽视、但最关键的部分。

### 18.1 必须严格控制 look-ahead bias

任何输入都必须满足“在事件时点真实可见”。

必须遵守：

- 新闻只允许使用 `publish_time <= decision_time`
- 价格特征只允许使用当前 bar 完结之前的数据
- 相似历史事件检索只允许使用事件当下可观测特征
- 不允许用未来收益率反向定义当前协变量

### 18.2 新闻时间戳要特别谨慎

对于新闻系统，尤其要区分：

- 新闻原始发布时间
- 页面抓取时间
- 入库时间
- 模型消费时间

建议把这四个时间全部保留，后续评估以“真实可交易时间”为准，而不是简单用入库时间代替。

### 18.3 历史分析结果不能直接回灌成当前特征

像“已被市场消化”“剩余定价空间”“主线标签”这类字段，很多是带未来信息的分析结论。

因此需要区分：

- 研究后标签：可用于复盘评估
- 决策时特征：只能用事件当下可得信息构造

### 18.4 支持样本检索要先定义相似性特征

未来做 ICF 时，support retrieval 是成败关键。

建议相似性特征只用以下可前视安全字段：

- 事件前 30m / 120m 收益率结构
- 事件前波动率
- 新闻密度
- 新闻类别标签
- 跨资产同步变化
- 时间段特征

不建议直接用“最终涨跌结果”去定义相似度。

### 18.5 分市场分层非常重要

建议不要把所有市场视为同一种时间序列任务。

#### 外汇

特点：

- 24 小时连续
- 宏观敏感
- 新闻驱动强

优先级：

- 首批优先资产

#### 黄金

特点：

- 受美元、利率、避险三重影响
- 适合事件驱动研究

优先级：

- 首批优先资产

#### 加密货币

特点：

- 周末不断档
- 波动大
- 新闻反应速度快

优先级：

- 可做首批，但要单独评估周末 regime

#### A 股与国内期货

特点：

- 有休市、跳空、交易时段断裂
- 数据连续性和交易可执行性更复杂

优先级：

- 放在第二阶段

## 19. 附录 D：模型监控、失效识别与降级策略

这部分决定系统能否真正生产化。

### 19.1 必须有模型健康度监控

建议监控以下指标：

- 最近 20 个交易日方向命中率
- 最近 20 个交易日 quantile coverage
- 不同资产上的稳定性
- 事件日上的表现变化
- 高波动时段表现变化

### 19.2 需要定义失效信号

建议把以下情况视为预警：

- 方向命中率持续低于 baseline
- 分位区间覆盖率明显失真
- 某一类事件上持续误判
- 模型分歧长期过高
- 预测服务延迟长期超阈值

### 19.3 必须有分级降级机制

#### 一级降级：停止交易用途

保留：

- 研究解释
- 风险提示

关闭：

- 交易信号输出

#### 二级降级：停止风险定量加权

保留：

- 方向参考

关闭：

- 概率直接参与仓位控制

#### 三级降级：完全退出主链路

回退到：

- 现有规则系统
- 历史样本统计系统
- LLM 风险经理文本判断

### 19.4 必须定义服务失败时的行为

当预测服务不可用时：

- 实时关注继续返回新闻与价格解释
- 风险经理退回规则版摘要
- 场景化路由不依赖预测字段
- 页面明确标记“预测层暂不可用”

### 19.5 必须定义上线门槛

建议 TimesFM 输出进入不同层级的门槛如下：

#### 进入研究层

- 只要求预测能稳定输出
- 允许误差较大

#### 进入风险层

- 要求 quantile coverage 稳定
- 要求高不确定性识别有效

#### 进入信号层

- 要求显著优于 baseline
- 要求手续费后仍有优势
- 要求跨多个资产和多个 regime 稳定

### 19.6 推荐的生产原则

- 先让模型做“研究增强”
- 再让模型做“风险增强”
- 最后才考虑做“交易增强”

这比一开始就直接做交易信号更稳妥，也更符合当前项目积累的优势。

## 22. 附录 G：里程碑与验收标准

这一部分用于让团队直接按阶段推进。

### 22.1 里程碑 M1：预测服务可运行

目标：

- 单资产单频率推理可用
- 返回统一结构化结果

验收标准：

- 能对 EURUSD 5m 数据成功返回预测
- 接口返回时间稳定
- 失败时能优雅报错

### 22.2 里程碑 M2：实时关注接入

目标：

- 实时关注中能显示预测方向与区间

验收标准：

- 前端可看到 30m / 120m 预测结果
- 页面不会因预测服务失败而崩溃
- 预测不可用时有明确降级提示

### 22.3 里程碑 M3：历史分析接入

目标：

- 每个价格事件都能生成事件时点预测

验收标准：

- 历史分析能显示预测与实际偏差
- 可输出 surprise score
- 不引入明显的时间泄露问题

### 22.4 里程碑 M4：风险经理增强

目标：

- 风险经理可使用分位区间与不确定性指标

验收标准：

- 高不确定性时能输出风险升级结论
- 方向明确但尾部风险大时能体现限制动作

### 22.5 里程碑 M5：离线评估完成

目标：

- 建立完整评估报告

验收标准：

- 具备 baseline 对照
- 具备分市场、分频率、分 regime 结果
- 具备事件日单独表现

### 22.6 里程碑 M6：进入研究层生产

目标：

- TimesFM 输出正式进入研究系统主链路

验收标准：

- 不影响现有接口稳定性
- 可稳定降级
- 研究员可在页面稳定消费结果

### 22.7 首期上线判定

只有满足以下条件，才建议首期正式上线：

- TimesFM 服务稳定
- 预测缓存稳定
- 方向与区间评估优于 baseline
- 页面链路具备降级能力
- 风险经理与历史分析均已接入

### 22.8 不建议首期承诺的结果

首期不建议对外承诺：

- 稳定交易收益
- 全市场全资产通用
- ICF 已工程化完成
- 模型可直接替代交易员判断

首期更合理的定位是：

- 研究增强工具
- 事件后概率辅助判断工具
- 风险过滤工具

## 23. 附录 H：Trae 多角色协作开发模式说明

这一部分用于回答一个实际工程问题：是否可以采用多角色协作方式推进 TimesFM 项目。

### 23.1 结论

可以。

更准确地说，Trae 在当前开发模式下非常适合采用“多角色协作式开发”，即：

- 用统一主控流程推进项目
- 用不同职责视角拆解需求
- 按前端、后端、测试、研究、运维等角色分工设计
- 最终由统一交付标准收敛

这不是简单地把项目拆给不同人各自开发，而是：

- 由一条主线规划任务
- 让不同角色在同一需求上从不同角度审视
- 通过统一接口和验收标准保证不割裂

### 23.2 为什么 TimesFM 项目适合多角色协作

TimesFM 项目天然横跨多个领域：

- 研究设计
- 时间序列建模
- 后端服务
- 前端展示
- 评估与测试
- 风控与降级

如果只从单一角色推进，容易出现以下问题：

- 后端能返回结果，但前端看不懂
- 前端展示很多字段，但没有稳定契约
- 模型能跑，但没有评估闭环
- 评估很好，但没有服务化与降级机制

因此更适合采用角色分层推进。

### 23.3 推荐的角色划分

建议至少按以下 6 个角色协作：

#### 角色 1：产品 / 研究负责人

职责：

- 明确研究目标
- 定义首期边界
- 定义哪些输出用于研究、风控、信号
- 定义优先资产与优先场景

核心交付：

- 研究目标说明
- 场景优先级
- 业务验收标准

#### 角色 2：量化研究负责人

职责：

- 设计标签
- 设计 baseline
- 设计评估框架
- 审核 TimesFM 是否真实产生增益

核心交付：

- 评估方案
- 基准模型对照
- 分层研究报告

#### 角色 3：后端负责人

职责：

- 设计 TimesFM 服务
- 设计输入输出契约
- 设计缓存与降级
- 接入 realtime focus / historical analysis / risk manager

核心交付：

- API 契约
- 预测服务
- 缓存与容错逻辑

#### 角色 4：前端负责人

职责：

- 设计预测结果展示方式
- 设计风险与不确定性展示
- 设计降级时的用户反馈
- 保证研究结果可读而不误导

核心交付：

- 预测卡片
- 风险带展示
- 降级状态展示

#### 角色 5：测试负责人

职责：

- 建立后端接口测试
- 建立前端展示测试
- 建立评估结果校验
- 建立回归测试与故障降级测试

核心交付：

- 单测
- 集成测试
- 前端验证清单
- 降级测试用例

#### 角色 6：平台 / 运维负责人

职责：

- 模型部署
- 推理资源规划
- 服务可用性监控
- 健康度与告警

核心交付：

- 部署策略
- 监控项
- 告警规则

## 24. 附录 I：前端、后端、测试统一开发流程

这一部分用于把多角色协作真正转成统一开发流程。

### 24.1 总体原则

采用“先定义研究契约，再并行开发前后端，最后统一测试收口”的流程。

不建议采用：

- 后端先写完再想前端怎么展示
- 前端先画页面再反推接口
- 模型先接入再考虑测试和降级

建议采用：

- 第一步统一定义输出契约
- 第二步后端实现与前端并行
- 第三步测试基于契约做自动校验

### 24.2 流程阶段

#### 阶段 1：研究定义

目标：

- 定义 TimesFM 输出到底服务什么

本阶段必须完成：

- 首期资产清单
- 首期频率清单
- 标签定义
- baseline 定义
- 上线门槛定义

负责人：

- 产品 / 研究负责人
- 量化研究负责人

#### 阶段 2：接口契约冻结

目标：

- 冻结前后端共同依赖的字段

本阶段必须完成：

- 预测接口字段
- realtime focus 新字段
- historical analysis 新字段
- risk manager 新字段
- 降级状态字段

负责人：

- 后端负责人
- 前端负责人
- 测试负责人共同确认

#### 阶段 3：后端能力开发

目标：

- 建立可调用的 TimesFM 服务与融合逻辑

工作项：

- 数据准备
- 模型推理
- 缓存
- 融合层
- 降级处理

输出：

- 稳定接口
- mock 数据
- 错误码定义

#### 阶段 4：前端展示开发

目标：

- 让用户能正确理解预测结果

工作项：

- 预测卡片
- 分位区间展示
- 高不确定性提示
- 预测层不可用提示

输出：

- 页面组件
- 展示规范
- 状态文案

#### 阶段 5：统一测试与回归

目标：

- 保证“研究正确、接口稳定、页面可读、降级可用”

工作项：

- 后端单测
- 前后端集成测试
- 前端展示校验
- 降级路径测试
- 历史评估结果核验

#### 阶段 6：灰度上线

目标：

- 先进入研究层，再进入风险层

建议路径：

- 仅内部可见
- 仅少量资产开启
- 只展示不做动作建议
- 评估稳定后再开放风险层引用

### 24.3 前端、后端、测试的统一接口思路

建议把每个预测字段都分成三层：

#### 原始层

用于机器消费：

- `point_forecast`
- `quantiles`
- `dispersion_score`

#### 结构化层

用于业务逻辑：

- `continuation_probability`
- `reversal_risk`
- `forecast_regime`
- `forecast_confidence`

#### 展示层

用于前端直接渲染：

- `summary`
- `risk_badge`
- `display_band`
- `degrade_message`

这样可以避免：

- 前端直接解释原始模型结果
- 后端反复写展示逻辑
- 测试无法判断到底测哪一层

### 24.4 推荐的前端交付标准

前端必须达到以下标准：

- 能区分“预测方向”与“风险区间”
- 能区分“研究结论”与“交易建议”
- 高不确定性时不误导用户
- 预测层不可用时有明确降级提示
- 支持实时关注、历史分析、市场总结三条链路统一展示语义

### 24.5 推荐的后端交付标准

后端必须达到以下标准：

- 接口字段稳定
- 输入窗口可复用
- 支持缓存
- 支持超时和降级
- 支持模型版本标记
- 支持评估与监控记录

### 24.6 推荐的测试交付标准

测试必须覆盖以下层面：

#### 后端单元测试

- 输入窗口构造
- 推理结果解析
- 融合逻辑
- 降级逻辑

#### 集成测试

- realtime focus 接口
- historical analysis 接口
- risk manager 融合结果

#### 前端验证

- 预测字段渲染
- 风险带展示
- 高不确定性状态
- 预测层故障提示

#### 研究测试

- baseline 对照结果
- 分市场结果
- 事件日结果
- Walk-forward 结果

## 25. 附录 J：TimesFM 项目审计说明书开发流程

这一部分用于指导“如何开发审计说明书”，而不是只开发代码。

### 25.1 审计说明书的目标

TimesFM 项目审计说明书不应只是技术介绍，而应回答以下问题：

- 为什么要做
- 现在做到哪一步
- 是否有真实研究增益
- 是否存在数据泄露风险
- 是否具备上线条件
- 前端、后端、测试是否统一收口

### 25.2 审计说明书建议分为五个部分

#### 第一部分：业务与研究目标审计

审计点：

- 目标是否清晰
- 是否区分研究层、风险层、信号层
- 是否定义首期边界

#### 第二部分：模型与数据审计

审计点：

- 数据源是否可靠
- 时间戳是否一致
- 是否存在 look-ahead bias
- 是否有 baseline 对照
- 是否有分市场评估

#### 第三部分：后端实现审计

审计点：

- 接口是否稳定
- 缓存是否合理
- 降级是否完整
- 是否支持模型版本化

#### 第四部分：前端交互审计

审计点：

- 是否正确表达不确定性
- 是否把研究信息误展示成交易建议
- 是否能展示降级状态
- 是否能统一多个页面的展示语义

#### 第五部分：测试与上线审计

审计点：

- 是否有回归测试
- 是否有评估结果输出
- 是否有灰度策略
- 是否有健康监控与回退策略

### 25.3 审计说明书开发顺序

建议按以下顺序开发：

#### 步骤 1：先审业务目标

输出：

- 目标边界说明
- 首期不做清单

#### 步骤 2：再审研究设计

输出：

- 标签说明
- baseline 清单
- 评估指标说明

#### 步骤 3：再审后端接口与服务

输出：

- API 契约
- 缓存设计
- 降级说明

#### 步骤 4：再审前端表达

输出：

- 展示规范
- 风险表达规范
- 高不确定性提示规范

#### 步骤 5：最后审测试与上线

输出：

- 测试覆盖矩阵
- 上线门槛
- 灰度策略

### 25.4 推荐的审计交付物

建议最终形成以下文档组合：

- 项目升级说明书
- 技术设计书
- 研究评估说明书
- 测试与验收说明书
- 上线与降级说明书

### 25.5 推荐的协作方式

建议由一个主控负责人统一收口，其他角色分别补充：

- 研究负责人补目标与评估
- 后端负责人补服务与接口
- 前端负责人补展示与交互
- 测试负责人补用例与验收
- 运维负责人补部署与监控

最终由主控负责人统一：

- 术语
- 字段定义
- 里程碑
- 上线判定

这样文档才不会出现“每部分都对，但整体不连贯”的问题。
